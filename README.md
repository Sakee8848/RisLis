请根据下面的核心设计交付文件，开发一个名字叫 **RisLis（中文名字叫：风团）** 的App。它涵盖了业务执行层（团长手册）、逻辑设计层（PRD）以及工程落地层（项目规划）。要求：
1）先从第一个“人身意外险”开始
2）假想接入的第一批保险公司包括：人保、平安、太平洋
3）Logo as attached
4）风格以简单直接明了为主基调，考虑选用苹果的风格，以User friendly为主要原则
5）多用户权限与界面，目前的role包括：
- 团长
- 新人
- 团员
- 保险公司
- 系统管理员

## 第一部分：RisLis 团长操作手册 (Captain's Manual)

### 1. 团长的使命与权力

作为团长，你不仅是管理者，更是团队的“首席风险官（CRO）”。你的目标是：**筛选掉那 20% 的高风险人员，为剩下的 80% 成员争取 50% 的保费降幅。**

### 2. 组团三部曲

* **第一步：设置准入门槛**
* 在 App 中选定你的团队属性（如：上海陆家嘴外卖团）。
* 订制保障包：建议首选“人身意外 + 工伤互助 + 骑行车险”。


* **第二步：发起 AI 风险背调**
* 向新人发送入团邀请链接。
* **强制动作：** 要求新人在线完成授权。AI 会实时抓取其交管违章、医保购药、历史赔付数据。


* **第三步：全员民主表决**
* AI 生成《风险报告》并给出 0-10 分的等级。
* **金律：** 报告自动推送到群组，全团成员必须在 24 小时内完成投票。**1 票反对即视为入团失败。**



### 3. 动态管理与退出机制

* **实时监控：** App 接入传感器数据（如超速、疲劳驾驶），团长后台会收到预警。
* **驱逐权：** 若成员发生严重违章或触发高额赔付，团长有权发起“强制退团”投票，保护全团明年的费率不被拖累。

## 第二部分：RisLis 产品需求文档 (PRD摘要)

### 1. 产品概述

* **名称：** RisLis (风团)
* **目标用户：** 蓝领领队（团长）、新经济从业者（团员）、保险公司。
* **核心价值：** 通过 AI 筛选与社群博弈降低逆向选择，消除保险佣金，回归保障本质。

### 2. 核心功能模块清单

| 模块 | 功能点 | 需求详述 |
| --- | --- | --- |
| **风险画像 (AI)** | 数据采集引擎 | 接入交管、医保、司法、征信等第三方 API 数据源。 |
|  | 风险评分模型 | AI 基于大数据算法生成 0-10 级评分（1级最安全）。 |
| **团管理 (DAO)** | 投票决策系统 | 采用全票通过制入团，动态公示团队综合赔付率。 |
|  | 规则编辑器 | 团长可自由增减险种、设定团队内部处罚规则。 |
| **保险接口** | 费率联动算法 | 接入保险公司接口，展示：。 |
| **身份体系** | 电子 ID & 视觉 | 自动生成带 Logo 和编号的电子卡片，关联线下贴纸申领。 |

### 3. 非功能性需求

* **隐私保护：** 采用**联邦学习（Federated Learning）**或数据脱敏技术，确保团长只能看到 AI 结论报告，无法查阅团员的病历原文。
* **隔离性：** 数据库采用多租户架构，确保各团数据完全独立。

## 第三部分：项目开发与实施规划

项目预计总周期：**24 周（约 6 个月）**，分为四个阶段。

### 阶段一：架构设计与数据合规 (Week 1-6)

* **重点：** 完成与第三方数据源（交管、医保等）的合规对接协议。
* **产出：** 数据库原型、AI 风险评分初步模型、合规白皮书。

### 阶段二：核心功能开发 (Week 7-14)

* **重点：** 1.  开发团长端、团员端双 App 界面。
2.  集成 AI 自动报告生成系统。
3.  开发投票系统及“全票通过”逻辑算法。
* **产出：** Beta 测试版 App。

### 3. 阶段三：保险接口与费率算法联调 (Week 15-20)

* **重点：** 1.  与首批合作保险公司进行 API 联调。
2.  实现“阶梯费率”自动结算逻辑（基于赔付率回传）。
3.  建立订阅计费系统（按人头计费）。
* **产出：** 具备投保能力的预发布版本。

### 4. 阶段四：实地试点与品牌宣发 (Week 21-24)

* **重点：** 1.  在北京/上海选定 10 个外卖/物流团进行压力测试。
2.  线下发放首批 RisLis 贴纸与 ID 卡。
3.  正式上线发布。

## 第四部分：风险预估与对策

* **法律风险：** 部分地区严禁“去中介化”保险变相经营。
* *对策：* 明确 RisLis 为**技术服务平台**，仅收订阅费，保险合同主体仍为成员与保险公司。


* **数据障碍：** 个别地区医保/交管接口不开放。
* *对策：* 引导用户通过 App 录屏或通过个人政务账号授权登录获取（由 AI 自动扫描解析）。

对于不同的工种、不同的地区、不同的年龄等这些“风险因子”，需要进行差异化定价，在保险公司的页面，需要：
1）留出多个可以增减的“风险因子”、“定价算法”的空格
2）给出一个Copilot的窗口，可以让核保自行输入、上传各类与“风险因子”以及相应算法有关的prompt or documents

同时，在团员各自的页面，需要：
1）给出相应“风险因子”的选择表，由团员自行选择维护相关数据并对自己的选择负法律责任（瞎填可能影响后续赔付以及风团整体打折系数）

对于“系统管理员”页面，需要加入对于不同保险公司的多角度评价/比较矩阵（从条款友好度、价格、理赔、品牌等等角度）
在“团长”页面里面，如果风险达到一个阀值以后，是不能轻易开除某人的：
1）首先折扣率与风团的历史赔付率挂钩，规定一个统计周期，由保司与客户沟通后事先确定，写进软件规则自动执行（这里不妨设定一个月），在此期间所有人的费率统一浮动；一个周期结束后再次计算赔付率重新调整保费折扣率；
2）当某位团员发生意外后，统计周期与赔付率超标浮动挂钩（超过一定阀值可以拒绝承保该风团），团员解散后可以申请加入其他团（但团队理赔记录跟随）；
3）需要在”团长“页面与“团员”页面上同时体现：
- 团队赔付率波动曲线（最近6个月）
- 费率波动曲线（最近6个月）
- 折扣率统计周期剩余日期
- 预计下一周期费率折扣率
